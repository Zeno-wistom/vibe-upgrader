# Real Project Acceptance

Test the two tracks separately. Use disposable targets or isolated routes; never modify a completed user project merely to test the Skill.

## Standard Scenario

Use a request focused on hierarchy, form usability, navigation, content, feedback, or responsive repair.

Pass when:

- `upgrade_track=standard`
- MotionSites and React Bits are skipped unless a specific creative gap requires them
- no experimental prototype is required for a scoped fix
- the result changes only authorized surfaces
- verification covers the affected task without a broad test matrix or long report

## Experimental Scenario

Use a request that explicitly asks for a strong visual, immersive, digital-art, or signature-interaction result.

Pass when:

- `upgrade_track=experimental`
- one to three references become one concrete mechanism brief
- the mechanism names its trigger, key states, spatial change, implementation primitives, and five-second target
- implementation stops at an isolated prototype before human approval
- prototype verification uses same-viewport before/after, one desktop viewport, one mobile viewport, build, and console
- full integration and expressive dependency installation remain blocked

## Rejected Component Scenario

Supply or retrieve an expressive candidate that fails stack, design-system, license, performance, or task-fit gates.

Pass when the output contains either:

- a custom-build fallback for the same expressive category and ambition, or
- an explicit human decision requesting permission to reduce the visual ambition

Reject any result that silently falls back to generic spacing, cards, fades, or ordinary layout polish.

## After Human Approval

Run a separate integration scenario. Verify only relevant real viewports, keyboard and focus behavior, reduced motion, runtime cost, original task flow, and regression boundaries. A passing build is not a substitute for visual acceptance or interaction acceptance.

## Cost Evidence

Record phase token usage when telemetry exists. Otherwise record the number of reference passes, prototypes, viewports, and reports. Fail the pre-approval scenario if it exceeds three references, one prototype, two viewports, or one report without a new user decision.
